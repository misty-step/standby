# Context Packet: Local-Capture Real Meeting Standby

## PRD Summary

- User: Phaedrus, using Standby during live calls on a Mac.
- Problem: Standby currently demonstrates the command-surface loop with seeded
  transcript spans and a mock research worker. The prior Vexa-first shape would
  make Teams work before it makes Standby work for any call on the computer.
- Goal: Make Standby listen to a real Teams call through local Mac audio capture
  and route an approved proposal to a real local worker agent with durable
  progress and artifact events.
- Why now: The core product promise is bot-free, any-call meeting work routing.
  App-specific capture adapters would solve the first Teams demo while pushing
  the actual product problem downstream.
- UX enabled: The user can start Standby for the active call, see honest
  microphone/system-audio/transcription status, approve an editable proposal,
  watch a real local agent job run, and inspect the resulting artifact or
  failure receipt.
- Deliverable type: Working code plus verification harness.
- Success signal: A Teams call playing through the Mac produces real transcript
  events from local audio capture, and an approved proposal produces an artifact
  generated by a local CLI agent, not `MockResearchWorker`.

## Product Requirements

- P0: Primary capture is `LocalMacAudioSource`: microphone plus system/app audio
  captured on the Mac. Teams is the first dogfood app, not a special first
  capture integration.
- P0: Add capture proof before product wiring. No implementation may claim
  live-readiness until a local capture smoke sees nonzero mic/system audio frames
  and records a sanitized evidence packet.
- P0: Transcription must be unstubbed. Use a real transcriber behind a
  `Transcriber` interface. On this machine, prefer Apple Speech because macOS
  26.5.1 and Swift 6.3.2 are available and no Whisper CLI is installed.
- P0: Real transcription needs a deterministic non-live proof. Generate a short
  local audio sample at test time, feed it through the same `Transcriber`, assert
  a final transcript segment, and delete the audio file after the run.
- P0: If native Apple APIs cannot be driven safely from Rust, use one tiny Swift
  helper process for ScreenCaptureKit/Speech only. Rust still owns durable state,
  API routes, event normalization, approval, job orchestration, and storage.
- P0: Approval must enqueue a worker job; it must not run long-lived worker CLI
  processes inside the HTTP approval request.
- P0: The first real worker is read-only research in an isolated scratch
  workspace with executable tool restrictions. Prefer a CLI profile that can
  deny shell/edit/write/MCP/send tools. Coding/repo mutation workers remain
  disabled until permission enforcement is executable.
- P0: Worker safety must be proven with negative tests. A malicious transcript
  fixture must fail to mutate the Standby repo, write outside job scratch, or
  trigger external-send tools, while still producing a visible failed or
  completed job event.
- P0: Transcript text remains untrusted evidence. It may create proposal cards
  with quoted evidence; it may not directly mutate repos, send messages, deploy,
  or spend money.
- P0: UI must distinguish demo, waiting for permission, capture active,
  no-system-audio, no-mic-audio, transcribing, failed, job queued, worker
  running, worker failed, and completed states.
- P0: Every source event, proposal, approval, job transition, artifact, and
  failure is appended to SQLite.
- P1: Keep Vexa/Recall/Teams/Zoom/Meet adapters as optional `TranscriptSource`
  providers for better speaker labels, bot capture, enterprise paths, or
  post-meeting backfill.
- P1: Preserve the current demo route as explicit demo mode, not as the default
  live UI path.
- Non-goals: Building a Microsoft Teams media bot first; making a Zoom/Meet
  adapter first; Microsoft Graph post-meeting transcript ingestion; mutation
  capable Codex/Claude repo jobs; automatic Slack/email/client messages;
  perfect speaker diarization in the first local-capture slice.

## Technical Design

Chosen architecture: keep Standby's durable core Rust-owned, and add three deep
modules behind narrow interfaces.

1. `TranscriptSource` adapter layer
   - First implementation: `LocalMacAudioSource`.
   - Inputs: meeting ID, selected capture mode, microphone device, system audio
     capture target, and local permission state.
   - Output: normalized `transcript.source.*`, `audio.source.*`, and
     `transcript.segment.partial/final` events.
   - Optional future implementations: `VexaBotSource`, `RecallBotSource`,
     `TeamsGraphImportSource`, `ZoomRtmsSource`, `MeetMediaSource`.

2. Native capture/transcription boundary
   - Preferred implementation path: Rust daemon supervises a small native helper
     process that owns ScreenCaptureKit and Apple Speech access.
   - The helper emits JSONL over stdout. It does not write SQLite, create
     proposals, run workers, send messages, or know worker credentials.
   - Event examples:

```json
{"type":"transcript.source.started","meeting_id":"mtg_...","source":"local_mac"}
{"type":"audio.source.level","lane":"microphone","rms":0.04,"captured_ms":1000}
{"type":"audio.source.level","lane":"system_audio","rms":0.11,"captured_ms":1000}
{"type":"transcript.segment.partial","speaker":"me","text":"let's research..."}
{"type":"transcript.segment.final","speaker":"system_audio","text":"can someone check what already exists?"}
{"type":"transcript.source.failed","reason":"screen_system_audio_permission_denied"}
```

3. `WorkerRunner` adapter layer
   - First implementation: local subprocess research worker.
   - Approval endpoint writes `proposal.approved` and `agent_job.requested`, then
     returns immediately.
   - A Tokio background worker or separate `standby-worker` process claims queued
     jobs, launches the configured CLI, streams normalized `agent_job.started`,
     `agent_job.progress`, `artifact.created`, `agent_job.completed`, or
     `agent_job.failed` events.
   - First profile: Claude Code read-only research, because local help confirms
     noninteractive mode, stream-json/json output, JSON schema support, tool
     allow/deny lists, permission modes, and budget limits. Pi is the fallback
     profile because local help confirms `--no-tools`, explicit read-only tool
     allowlists, JSON mode, and no-session mode. Codex remains the later coding
     worker target unless a Codex profile passes the same sandbox verifier.
   - Candidate command shape:

```text
claude -p \
  --output-format stream-json \
  --json-schema .standby/jobs/<job_id>/output.schema.json \
  --max-budget-usd 1.00 \
  --disallowedTools Bash Edit Write \
  --no-session-persistence \
  "<approved prompt plus a data envelope of quoted transcript evidence>"
```

   - The scratch workspace is the only writable target for this worker profile.
     The Standby repo and external messaging systems are not mounted as writable
     worker context in this slice. If the installed CLI cannot enforce the
     declared restrictions, `verify-worker-sandbox.sh` must fail and the profile
     is not accepted. The runner should launch the profile with no MCP config and
     the smallest environment that still permits model auth.

Data/control flow:

```text
Teams call audio routed through Mac
  -> LocalMacAudioSource
  -> native capture/transcription helper
  -> normalized transcript events
  -> proposal detector
  -> proposal card
  -> deterministic approval endpoint
  -> queued AgentJobSpec
  -> WorkerRunner subprocess
  -> normalized job/artifact events
  -> React card projection
```

Build/check boundary:

- Build catches schema mismatches, route wiring, and TypeScript projection
  expectations.
- Capture smoke catches macOS permissions, device routing, and whether mic and
  system audio produce nonzero frames.
- Transcript fixture replay catches ordering, partial/final behavior, proposal
  evidence requirements, and projection stability without needing a live call.
- Live Teams local-capture smoke catches actual Teams audio routing and
  transcription flow.
- Worker smoke catches CLI availability, auth prompts, timeout behavior, scratch
  workspace isolation, and artifact persistence.
- Worker sandbox smoke catches malicious transcript injection and verifies the
  selected CLI profile cannot mutate repo files or call external-send tools.

ADR decision: required after this slice, not before. This slice can live as a
context-packet-backed implementation. Create an ADR if a second capture provider
or mutation-capable worker is added, because that commits the adapter and
permission model as product architecture.

ADR-style invariants:

- Invariant: approval is deterministic and server-owned. If violated, transcript
  text can become an LLM-controlled action path. Core refs:
  `AGENTS.md`, `crates/standbyd/src/main.rs`.
- Invariant: local capture is the default product path. If violated, Standby
  becomes a per-meeting-app adapter project instead of an any-call command
  surface. Core refs: `docs/research/real-meeting-capture-and-workers.md`,
  `crates/standby-core/src/domain.rs`.
- Invariant: provider-specific capture stays behind transcript source adapters.
  If violated, Vexa/Teams/Zoom/Meet details leak into proposal/UI/job modules.
  Core refs: `crates/standby-core/src/domain.rs`,
  `crates/standby-core/src/engine.rs`.
- Invariant: worker execution is out-of-request and event-sourced. If violated,
  the UI hangs and failures disappear. Core refs:
  `crates/standby-core/src/worker.rs`, `crates/standby-core/src/event_log.rs`.
- Invariant: first real worker cannot mutate repos or external systems. If
  violated, an approved meeting card can have a larger blast radius than the
  user saw in the card. Core refs: `crates/standby-core/src/worker.rs`, future
  worker-runner verifier scripts.

## Alternatives

| Option | Why it helps | Failure mode | Verdict |
| --- | --- | --- | --- |
| Local Mac capture first | Matches the actual product promise: works with Teams today and any call routed through the computer later. Mirrors Granola/Meetily/Minutes category pattern. | macOS TCC permissions, device-routing edge cases, weaker speaker identity. | Choose. |
| Vexa Teams adapter first | Fast Teams transcript path with bot and WebSocket protocol. | Solves Teams before solving any-call; bot admission/vendor/account risk; pushes local capture later. | Defer to optional provider adapter. |
| Microsoft real-time media bot first | Platform-native raw media and Teams official support. | Windows/.NET-shaped and specialized; wrong first path for a local Rust/Mac product. | Reject for first slice. |
| Microsoft Graph transcripts/AI insights first | Official Graph permissions and post-meeting artifacts. | Not live; insights can be delayed and may require licensing. | Reject for live surface; useful later for import/backfill. |
| Manual transcript paste first | Boring path, no capture complexity. | Does not help during the meeting and cannot prove proposal timing, permissions, or source state. | Reject as primary; keep fixture replay for tests. |
| Keep mock worker and only fix capture | Easier and narrows implementation. | Violates user requirement; approved cards still do not do real work. | Reject. |
| Launch CLI worker synchronously in approval request | Smallest diff. | Hangs UI, loses progress, auth prompts block HTTP, no durable retries. | Reject. |

Delete-first / Ponytail answer:

- Requirement questioned: do we need one adapter per app for the product to hear
  meetings? No; the outcome is audio from the active call on the Mac.
- Deleted or simplified: no Teams media bot first, no Vexa-first dependency, no
  Zoom/Meet adapter first, no mutation-capable worker marketplace, no perfect
  diarization in slice one.
- Only then optimized/automated because: local capture plus one real read-only
  worker is the shortest path that satisfies any-call input and real approved
  work.

## Oracle

Commands that must exist and exit 0 after implementation:

- `./scripts/verify.sh` - existing Rust, frontend, build, and API smoke remain
  green.
- `./scripts/verify-local-capture-smoke.sh` - runs the local capture helper,
  verifies microphone and system-audio lanes produce nonzero frame/level events,
  and writes sanitized evidence under `docs/evidence/real-meeting/`.
- `./scripts/verify-real-transcriber-smoke.sh` - generates a temporary local
  audio sample containing a known phrase, feeds it through the real transcriber,
  asserts a final transcript segment contains the expected words, and deletes the
  audio file.
- `./scripts/verify-local-transcript-fixture.sh` - replays transcript events
  shaped like local capture output, verifies partial/final ordering, proposal
  evidence citation, dedupe, and projection stability.
- `./scripts/verify-worker-runner.sh` - enqueues a read-only research job,
  launches a real installed CLI in a bounded scratch workspace, records stdout or
  final response to an artifact file, and appends completed/failed events.
- `./scripts/verify-worker-sandbox.sh` - runs a malicious transcript/job fixture
  that tries to edit the Standby repo, write outside `.standby/jobs/<job_id>`,
  and invoke external-send behavior; passes only if those attempts are denied or
  impossible and the denial is evented.
- `./scripts/verify-ui-states.sh` - drives the local UI against fixture events
  and asserts demo is not auto-started on the normal route; waiting-permission,
  no-audio, transcribing, worker-running, worker-failed, and completed states
  render with screenshots saved to `docs/evidence/real-meeting/`.
- `STANDBY_LIVE_CAPTURE=1 ./scripts/verify-live-teams-local.sh` - when the
  operator can join a Teams test call, starts local capture, receives at least
  one final transcript segment from Teams audio, creates a proposal, approves it,
  and observes a real local worker artifact.

Observable outcomes:

- The UI no longer auto-starts demo mode on the normal meeting route.
- Source status is explicit: demo, waiting for permission, capture active,
  no-system-audio, no-mic-audio, transcribing, failed, or stopped.
- A denied macOS permission produces a visible failure card and
  `transcript.source.failed` event.
- A silent Teams/system audio route produces a visible no-audio state instead of
  an empty "live" transcript.
- A failed CLI auth or timeout produces a visible `agent_job.failed` card with a
  receipt path, not a spinner.
- Approved jobs show live status from event projection, not hard-coded progress.

## Verification System

- Claim: Standby can be used in a real Teams meeting on this Mac to detect a
  useful proposal and run approved read-only research through a local agent.
- Falsifier: a demo fixture can pass while macOS capture lacks permission,
  system audio is silent, transcription is stubbed, proposal evidence is missing,
  malicious transcript text mutates files, or worker CLI auth hangs the approval
  request.
- Driver: local capture smoke, transcript fixture replay, real Teams local
  capture smoke, deterministic real-transcriber smoke, worker sandbox smoke,
  local worker smoke, browser/UI state verification, and `./scripts/verify.sh`.
- Grader: event log contains expected source/audio/transcript/job/artifact event
  types; projection has ordered transcript segments; generated audio produces a
  final transcript segment; worker artifact file exists; worker cannot write
  outside job scratch; HTTP approval returns before worker completion; UI shows
  correct source/job states.
- Evidence packet: `docs/evidence/real-meeting/` should contain sanitized
  capture logs, transcriber smoke output, transcript fixture replay output,
  worker sandbox receipts, worker artifact, browser screenshots, and final event
  projection JSON.
- Cadence: build capture smoke first; run fixture and worker scripts before each
  milestone; run live Teams local smoke before claiming dogfood readiness; run
  browser check before closeout.
- Gaps / waiver: local capture requires macOS TCC permissions and a meeting/call
  where Teams audio is routed through the Mac. If unavailable, implementation is
  fixture-ready or capture-smoke-ready only, not dogfood-ready.

## Risks + Rollout

- macOS permission friction: Microphone and Screen & System Audio Recording may
  require manual approval. Mitigation: permission-specific failure events and
  capture smoke that reports the exact missing permission.
- Audio routing mismatch: Teams can use a device different from the system
  default. Mitigation: show mic/system-audio lane activity and fail with
  no-audio state when a lane is silent.
- Speaker attribution weakness: local capture may only know `me` versus
  `system_audio`. Mitigation: labels stay honest; provider adapters can add
  speaker diarization later.
- Native boundary creep: Swift code can grow into product logic. Mitigation:
  helper only emits JSONL capture/transcript events; Rust owns all durable
  behavior.
- CLI auth prompts: local worker commands can block. Mitigation: process
  timeout, stderr capture, and `agent_job.failed`.
- Proposal quality: current detector is demo-specific. Mitigation: first slice
  adds real transcript-shaped fixtures and requires evidence-cited proposal
  tests.
- Permission drift: future coding workers can mutate files. Mitigation: keep
  first worker read-only/scratch-only and require an ADR plus executable
  permission tests before enabling mutation-capable profiles.

Stop conditions:

- The capture helper cannot obtain microphone or system-audio permission; report
  permission-blocked, not live-ready.
- The capture smoke cannot observe nonzero frames from both mic and system audio;
  report capture-blocked and do not proceed to live Teams claims.
- The transcriber cannot emit real final segments from captured audio; report
  capture-ready but transcript-blocked.
- The worker runner cannot launch any installed CLI without interactive auth;
  report capture/transcript-ready only.
- The selected worker profile cannot pass malicious transcript and scratch-only
  negative tests; report worker-blocked and do not expose approval-to-worker UI.
- The UI cannot show source/job failures distinctly; do not claim usable live
  meeting behavior.

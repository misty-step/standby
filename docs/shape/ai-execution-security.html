PLAN / STANDBY

008 secure approval and AI execution

BACKLOG 008 / CONTROL PLANE BEFORE TOOL WORKERS

Only the local operator can approve AI work, and network workers need a per-job consent event before dispatch.

Chosen design: keep read-only projection open, protect every local mutation with a server-minted operator token plus same-origin checks, bind approval identity server-side, and make the worker runner independently enforce network consent plus prompt redaction.

QUALITY SYSTEM

Standards: local-first, Rust-owned policy, append-only events, no transcript-as-instruction, no default network worker.

Proof: adversarial HTTP replay, worker policy tests, full repo gate, and security critic focused on spoofed approvals or silent exfiltration.

STOP RULES

Stop if the normal UI cannot approve same-origin work, if a hostile origin can approve, or if a network worker can start without consent.

Premise source: backlog.d/008-secure-approval-and-ai-execution-gate.md

AUTH

Mutation routes require operator token via same-origin cookie or explicit local CLI header.

ORIGIN

Browser POSTs must come from the same host that served Standby; permissive CORS is removed.

APPROVAL

Approval actor comes from server config, not request JSON.

WORKERS

Network-capable jobs need a matching consent event and redacted prompt before launch.

NON-GOAL

No OpenCode/GLM/MCP default worker in this slice; this gates that work.

DRIVER

scripts/verify-ai-execution-security.sh starts the daemon, replays unauthenticated, hostile-origin, authenticated, spoofed-actor, and network-worker cases.

GRADER

HTTP status codes, event payload assertions, job terminal states, consent event checks, and redaction fixtures.

EVIDENCE

docs/evidence/ai-execution-security/ stores request/response receipts and a verdict JSON.

PHASES

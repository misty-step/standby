Deadlock-Free, Output-Independent Capture Helper

Two architectural bugs found in live dogfood that the milestone smokes missed. Reproduced and root-caused before shaping.

The work contract

Outcome: Standby captures mic + all participants for a full real meeting on any output device, never deadlocks, stops cleanly on SIGTERM, and names the exact missing macOS permission.

Chosen design — two pillars:

1 · Concurrency rewrite — @main async (delete dispatchMain()), bounded AsyncStream handoff from realtime threads, one non-blocking stdout writer, structured TaskGroup shutdown.
2 · Core Audio Process Taps — AudioHardwareCreateProcessTap + aggregate device for output-independent system audio; kTCCServiceAudioCapture tier; signed .app.
3 · Proof — a ≥60s daemon-driven capture smoke that reproduces the deadlock, kept red until fixed.

Why it wins

  • Kills the root cause (GCD vs Swift-executor fight), not the symptom.
  • Output-device-independent: solves HDMI/Bluetooth where ScreenCaptureKit yields zero frames.
  • Realtime audio threads never lock or do I/O — no second freeze path.

Stop conditions

  • Long-run smoke still plateaus after the rewrite → deadlock model wrong; re-investigate before the tap.
  • Can't get kTCCServiceAudioCapture → ship concurrency fix + ScreenCaptureKit (built-in), gate the tap.
  • A framework genuinely needs a main CFRunLoop → document + choose run-loop-safe structure, never reintroduce dispatchMain() with main-actor deps.

Current state — the reproduced failure

Deadlock

t=1..22s mic_level_events=0 · SIGTERM → STILL ALIVE (kill -9 only)

Reproduces in mic-only, via daemon/pipe/file. SIGTERM-immune = the main GCD queue is wedged. --seconds smokes raced the wedge and won — the gap.

No system audio on HDMI

Default output = HDMI; ScreenCaptureKit capturesAudio delivered 0 frames even with audio playing. Mic worked (then froze at the deadlock).

Root cause

dispatchMain() parks the one main thread on the GCD queue and never pumps Swift's main-actor executor → AVFoundation/Speech continuations that hop to main never resume. The audio render thread mints an unbounded Task { await pipeline.feed } per callback → cooperative-pool flood. Secondary: LevelMeter.observe does queue.sync { emit() } with a blocking FileHandle.write on the realtime audio thread.

Risks & review outcome

Fresh-context critique done — concurrency core verified sound (Core Audio IOProcs + HAL listeners take dispatch queues; no CFRunLoop survives @main async). 9 amendments folded into the packet. The 3 that were production-embarrassing:
  • 60s gate ≠ 10-min claim — aggregate-device clock drift hits at 20–40 min → added verify-capture-meeting-duration.sh (≥10 min) as the ship gate + drift correction on every sub-tap.
  • Ad-hoc signing evaporates the TCC grant on rebuild → require a stable signing identity + a verify.sh preflight that rejects ad-hoc.
  • No public pre-check for kTCCServiceAudioCapture → detect permission by attempt-and-classify, not a pre-emptive read.
Plus: muteBehavior must keep the call audible; transcriber-bound audio drops must be counted/emitted (never silent); per-PID tap to limit mic bleed; explicit AVAudioConverter for tap format.

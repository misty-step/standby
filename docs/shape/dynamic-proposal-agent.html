Context Packet — Dynamic Proposal Agent

Deliverable: code + eval · supersedes the "go OpenAI Realtime" instinct (research killed it) · turns 011 real · Shaped 2026-06-23

Target outcome: proposal cards become genuinely contextual — they appear, update, and retire as the live conversation shifts (talk market research → the card becomes about market research) — driven by a real model the first time, never a stub.

Chosen design
A real OpenRouter reasoner (default, not a stub) called on a debounced/changed-transcript trigger (not per-segment) over a sliding window, returning ledger patch ops (add / update / retire by card id) against an event-sourced card ledger. Strict JSON via response_format + require_parameters; honest failure, no keyword fallback.
Why it wins
Fixes the two real causes the operator hit live: (1) the default was a content-blind fixture; (2) per-segment firing + no supersede = spammy and frozen. Cascade beats Realtime for a non-speaking listener (5–30× cheaper, strict-JSON-capable, no 60-min cap). ~$0.08–0.17/meeting.
Proof surface
011 eval over a labeled corpus using recorded real OpenRouter responses (not a stub) + a gated live run; live-QA: a real meeting where pivoting topics visibly updates/retires the card.
Stop conditions
(1) z-ai/glm-5.2 not on OpenRouter's structured_outputs list → use deepseek-v4-pro, don't hack around strict. (2) Reasoner can't reliably emit retire/update ops in eval → escalate to gpt-5.4-mini before adding a gate. (3) If the schema/ledger change ripples into approval/worker dispatch unexpectedly → stop, the seam is wrong.

Goal · Non-Goals · Constraints

Goal

Make live proposal cards reflect what is actually being discussed and stay current as the conversation moves — produced by a real model by default, with reliable structured output and honest failure.

Non-Goals

  • No OpenAI Realtime API. Research (this session) shows it's the wrong tool for a non-speaking listener: 5–30× cost, no structured outputs (Agents SDK: "RealtimeAgent does not support structured outputs"), weaker tool-calling, 60-min cap + latency drift. We already have a free on-device transcript.
  • No tool/function-calling for cognition. The card/ledger patch is the model's answer, validated and applied by deterministic Rust → response_format, not tools. (Red Line: the server owns mutation; the model suggests only; no transcript path gets tool/network surface.)
  • No stub-as-default. Recorded becomes test/fixture-only. Ever.
  • Not in this slice: a separate cheap "gate" model, semantic-embedding dedup, and multi-card concurrency. Deferred to a measured fast-follow (see Alternatives B).
  • No UI redesign (the UX-polish items live in 013); this is the cognition engine.

Constraints / invariants

  • Rust owns the ledger; every card add/update/retire is an append-only event (the projection is the source of truth). Cognition never mutates external systems.
  • Evidence-cited: every card cites real transcript segment ids/indexes (the existing candidate_evidence rejection at engine.rs:641-676 stays).
  • ./scripts/verify.sh stays green; verify-model-proposals.sh's "no heuristic engine symbols" grep stays green (we add a model provider, we don't add keyword logic).
  • Old ledgers must still project (event-sourced: new event types are additive; ProposalStatus gains a variant — keep serde tolerant).
  • Approved cards are immutable and dispatch their current content. No update/retire may alter an Approved card (fold-enforced), and approve/ignore must resolve the latest (post-update) payload, not the original.

Premise challenge & alternatives

The request was "make it dynamic with a gate + reasoner + dedup." Challenged: the operator's real outcome is "cards track the conversation and don't freeze." The cheap-gate and embedding-dedup are cost/quality optimizations, not prerequisites — a debounced reasoner that patches a small ledger delivers the outcome with far less surface. Build the outcome; add the optimizations when data demands.

Design

1 · Provider boundary — ProposalProvider::OpenRouter (direct, not via OpenCode)

Add a third arm to the existing enum (engine.rs:26-34), mirroring the OpenAi arm's thread-spawn + blocking reqwest plumbing but posting to OpenRouter Chat Completions (https://openrouter.ai/api/v1/chat/completions) instead of the Responses API. Reuse proposal_agent_instructions(), proposal_agent_input(), and the parse_model_response → ModelProposalResponse tail verbatim (engine.rs:442-446). Cognition stays a thin read-only API call — never the OpenCode worker (that's sandboxed post-approval execution; routing cognition through it inverts the trust model and adds subprocess+sandbox latency for a per-segment call).

// from_env(): real model is the DEFAULT. Recorded is test-only.
match STANDBY_PROPOSAL_PROVIDER (default "openrouter"):
  "openrouter" => OpenRouter { api_key: OPENROUTER_API_KEY,
                               model:  STANDBY_OPENROUTER_PROPOSAL_MODEL  (default "deepseek/deepseek-v4-pro") }
  "openai"     => OpenAi { ... }            // kept
  "recorded"   => Recorded { ... }          // TEST/FIXTURE ONLY — never the default

// request body (Chat Completions):
{ model, messages:[{role:system, instructions}, {role:user, input}],
  response_format: { type:"json_schema", json_schema:{ name, strict:true, schema: proposal_response_schema() }},
  provider: { require_parameters: true, sort: "latency" },   // <- only route to providers that honor strict
  plugins: [ { id: "response-healing" } ], temperature: 0 }
Reliable strict JSON, no stub fallback (verified vs OpenRouter docs, 2026-06-23): provider.require_parameters:true is load-bearing — without it OpenRouter routes to a provider that silently drops the schema and returns prose (or a misleading 404). Response-Healing repairs JSON syntax (not schema). Then serde parse is the schema validator: on failure, retry once with a corrective system message; on exhausted retries emit the existing honest no_proposal("model_provider_error: …") (engine.rs:178-194) — not a keyword card. Verify z-ai/glm-5.2 appears under openrouter.ai/models?supported_parameters=structured_outputs before pinning it; otherwise default to deepseek-v4-pro.

2 · Trigger & cadence — decouple from per-segment

Today ingest() calls the agent on every finalized segment (transcript_source.rs:297-302) — the spam/freeze root. Replace with a debounced, change-gated trigger owned by the daemon/capture supervisor: coalesce to at most one reasoner call per N seconds (default ~15–20s, env-tunable) and only when the transcript actually advanced since the last call. The reasoner receives a sliding window (recent K segments + the current open cards) — not the whole transcript — keeping cost flat and the model focused on "now". (A separate cheap classifier gate is Option B; the debounce + change-gate is the simpler trigger that ships first.)

3 · Card ledger — event-sourced patch ops

The reasoner returns operations against the open cards, so a restated topic updates the open card and a dropped topic retires it — instead of freezing one card or stacking duplicates. This extends the proposal data model (today single-card, ProposalStatus = Proposed|Approved|Ignored). The six points below are load-bearing — a fresh-context critic proved the naive version dispatches stale cards, lets the model clobber approved ones, and re-creates the duplicates it exists to kill:

  • Approve/ignore must act on the CURRENT card (blocker). find_latest_proposal (event_log.rs:373-379) scans only proposal.created/approved/ignored — it is BLIND to proposal.updated. After a card updates ("→ market research"), the operator approves it in the UI and the server would dispatch the pre-update payload (stale title/prompt/evidence via build_job_spec, worker.rs:100-117). FIX: add PROPOSAL_UPDATED/PROPOSAL_RETIRED to that query's event-type list. (The earlier "approval paths untouched" assumption WAS the bug.)
  • Approved cards are immutable — enforced in the fold, not the prompt (blocker). upsert_by_id (event_log.rs:458-467) blind-replaces by id with no status guard, so a retire/update op targeting an Approved card would clobber it. Add a status check before replace that DROPS any update/retire whose target is already Approved. A prompt-only "don't touch approved cards" is not enforcement against a non-deterministic open model. (In-flight worker jobs are independently safe — dispatch + recoverable_jobs key off agent_job.*, never proposal status — but the projected card state must not be clobberable.)
  • Closed id contract or it re-creates duplicates (major). The server mints ids today (new_id("prop"), engine.rs:217-218) — the model has never seen one, and proposal_agent_input() (engine.rs:460-476) passes transcript only. FIX: inject each open card's real id into the model input, AND have the fold reject any update/retire whose card_id isn't a currently-open card (mirror the unknown-segment-id rejection in candidate_evidence, engine.rs:653-657). Without this, a hallucinated id misses upsert_by_id and falls through to push a new orphan card.
  • Deterministic post-parse op-coherence validator — NOT serde alone (major). Strict JSON Schema can't express "card_id required for update/retire, absent for add" in a flat object, and serde's pervasive #[serde(default)] (engine.rs:63-74,83-86) silently accepts a retire with no id. Model the schema as a per-op discriminated union, and regardless, validate every op after parse (op↔card_id coherence; card_id resolves to an open card; evidence passes candidate_evidence). On ANY violation → honest no_proposal("model_provider_error: …"), never a silent partial mutation. "serde parse is the schema validator" is false for these cross-field constraints.
  • Events: add PROPOSAL_UPDATED + PROPOSAL_RETIRED (domain.rs:475-479) + ProposalStatus::Retired (domain.rs:47); fold via the existing upsert_by_id arm (event_log.rs:225-229) plus the immutability guard above. A retired un-approved card drops out of the open set + the open-proposal gate.
  • Dedup (this slice): with open-card ids in the input, the reasoner is instructed to update rather than add when the topic matches — model-driven dedup. Embedding-similarity dedup is Option B.

Build sequence (verification system first)

  1. Provider + real default first. Add ProposalProvider::OpenRouter, flip from_env() default to it, demote Recorded to test-only. Prove a real card from a live transcript end-to-end (the operator can re-test the Meet immediately). Update BOTH gates: verify-model-proposals.sh's fixture-shape check (now {operations:[…]}, not proposals[]) and verify-live-model-proposal.sh (today pins STANDBY_PROPOSAL_PROVIDER=openai + asserts provider==openai — repoint to openrouter). Keep the recorded()/recorded_fixture() test constructors; CI fixtures become recorded real responses, not the synthetic stub.
  2. Ledger patch ops. Extend schema + add PROPOSAL_UPDATED/RETIRED + ProposalStatus::Retired + projection fold. Reasoner emits add/update/retire over open cards.
  3. Debounced/change-gated trigger replacing per-segment firing; sliding window.
  4. 011 eval wired (the proof): the corpus runs through recorded real responses (CI anchor) + gated live (STANDBY_LIVE_MODEL=1), scoring whether topic-pivots produce the right update/retire ops. Cost guardrail + honest-failure path land here.

Oracle — executable definition of done

  • With OPENROUTER_API_KEY set and no provider override, a fresh meeting produces a card whose content reflects the actual transcript (provider in the projection = openrouter, not recorded-model).
  • Eval: a corpus case that pivots topic (e.g. action-items → market-research) yields a retire/update op so the open card tracks the pivot; a deliberately content-blind output fails the eval.
  • Strict-JSON: a forced malformed/contractless response triggers exactly one retry then an honest no_proposal("model_provider_error") — never a keyword card. (Test with a fixture / a bad model.)
  • Regression (the critic's blockers): (a) approve a card, then a retire/update op targeting it leaves it Approved + unchanged (fold guard); (b) approving a card AFTER an update dispatches the worker from the UPDATED payload, not the original (find_latest_proposal sees proposal.updated); (c) an update/retire with a card_id not in the open set → honest no_proposal, never a new orphan card.
  • ./scripts/verify.sh green; verify-model-proposals.sh heuristic-symbol grep green; old ledgers with only proposal.created still project.
  • Per-meeting model spend stays under a configured ceiling (gate-rate/debounce cap honored).

Risks + rollout

  • Open-model strict-JSON adherence wobbles. Mitigated by require_parameters + healing + validate-retry + the gpt-5.4-mini fallback. Stop-condition if eval shows it can't hold the contract.
  • Ledger change ripples into approval/worker dispatch. Keep approve/ignore paths untouched; retire only affects un-approved cards. Regression test: approve a card, then a retire op must NOT un-approve it.
  • Cost surprise. Debounce + sliding window + per-meeting ceiling; the eval reports real per-meeting spend.
  • Op-thrash on a flapping conversation (topic A→B→A → retire/add churn). The spend ceiling is not a stability damper — add hysteresis (don't retire-then-re-add the same card within a window); 011 must measure churn, not only decision quality.
  • Keyless bare runs regress. Today a keyless cargo run -p standbyd shows a (content-blind) card; with the real default it shows only no_proposal until OPENROUTER_API_KEY is set. Make that an explicit, actionable first-run state ("set OPENROUTER_API_KEY"), honest — not silently empty.
  • Rollout/undo: additive (new provider arm, new event types, new status variant, trigger swap). Revert = restore the per-segment call + default provider; new event types are forward-compatible in the ledger.

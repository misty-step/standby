Backlog 005

Live speaker attribution without fake names

Standby now has a Rust-owned diarization/provider seam. Generic provider labels map into stable remote buckets, then the existing event log, proposal evidence, and UI rendering paths carry those buckets forward.

Acceptance

Fixture-backed provider attribution emits at least two remote speakers, renders them as Speaker 1 and Speaker 2, and prevents raw provider labels from leaking into the UI.

Chosen Slice

Add DiarizationProvider and diarization.segment.final JSONL events. Normalize generic zero-based acoustic labels into remote_N; provider-specific adapters must emit explicit remote_N when their labels use a different scheme.

Boundary

The Swift helper remains capture-only. Provider credentials, buffering, naming, and diarization orchestration stay outside the helper.

Limit

No acoustic diarizer is bundled yet. This proves the product contract for provider and sidecar outputs.

System Flow

Provider Emits remote acoustic buckets such as SPEAKER_00, SPEAKER_01 transcript spans.
Rust Seam Maps generic labels to remote_1, remote_2.
Event Log Stores ordinary transcript segment events.
Projection Carries speaker buckets into proposals and UI state.
UI Renders Speaker 1, Speaker 2, no fake names.

Proof

scripts/verify-live-speaker-attribution.sh runs the fixture through the daemon seed endpoint, checks projection speakers and proposal evidence, dumps the rendered DOM, captures a screenshot, and writes docs/evidence/operator-action-control/live-speaker-attribution/verdict.json.

Residual risk: true live acoustic diarization still requires an OpenAI, ElevenLabs, pyannote, or NVIDIA adapter behind this seam.

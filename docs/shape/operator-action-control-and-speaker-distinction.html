PLAN / STANDBY

operator action control + speaker attribution

CONTEXT PACKET / LIVE TEAMS DOGFOOD FOLLOW-UP

Give the operator an intentional proposal prompt and stop flattening multi-person calls into one speaker.

Chosen design: add an Ask Standby request path that creates evidence-cited proposal cards, preserve explicit approval before any job runs, and add a speaker-attribution seam so remote participants can become stable labels instead of one undifferentiated system-audio bucket.

CHOSEN DESIGN

First: operator prompt -> proposal request event -> one to three proposal cards -> existing approval/job/report path.

Second: speaker attribution seam with a verified source of stable remote speaker keys.

Updated: worker-profile planning is superseded by the default OpenCode subagent worker in ADR 0002 and backlog 009.

PROOF

verify-manual-proposal-request.sh, verify-speaker-distinction-fixture.sh, full ./scripts/verify.sh, and gated live Teams QA.

ACCEPTANCE

Typed operator request creates proposal cards with cited transcript context; approval runs a visible worker job with receipt.

SPEAKERS

A fixture with multiple remote speakers does not collapse to a single system_audio label.

SAFETY

No transcript or chat text executes directly. Approval queues the single OpenCode worker; no harness selector or fallback path.

STOP IF

No executable oracle, no stable speaker key source, or worker profile needs broad network/tools without proof.

CURRENT STATE

Teams live capture transcribes locally. Projection speaker labels in the observed Teams run were only system_audio.

CHANGE SHAPE

Add one operator request route, one UI input/control surface, evented proposal-request data, and speaker-attribution projection support.

NOT IN FIRST CUT

No OMP/GLM/MCP profile path, no auto-execution, no fake participant names, no raw transcript/audio persistence.

REPO ANCHORS

crates/standby-core/src/engine.rs - proposal generation and cited evidence.

crates/standby-core/src/domain.rs - proposal, worker, transcript, event schemas.

crates/standby-core/src/transcript_source.rs - helper speaker labels and normalization.

crates/standby-core/src/worker.rs - sandboxed worker profiles.

crates/standbyd/src/main.rs and ui/src/main.tsx - routes and operator surface.

INVARIANTS

Approval stays deterministic and server-owned. Transcript text remains evidence, not instruction. Every proposal request, proposal, job transition, artifact, and failure is evented. Labels must state uncertainty.

Context Packet — Proposal-Quality Eval Harness

Ticket backlog.d/011-proposal-quality-eval-harness.md · P0 · keystone · Deliverable: code + eval report · Shaped 2026-06-22

Target outcome: turn the AI-first proposal bet from asserted into a measured, gated number — replace "a card was produced" with precision / recall / false-proposal-rate over a held-out, labeled transcript corpus, with a deterministic CI anchor and a confidence-bounded live measurement.

Chosen design
A labeled corpus drives ProposalAgent::propose() directly. Deterministic decision metrics (propose-vs-silent, kind, gold-evidence overlap) — no LLM-judge in v1. CI replays per-case recorded real-model responses (a regression anchor); a gated live run re-scores the real model K times and reports a bootstrap CI.
Why it wins
The decision metric is the bet ("asks → cards, chatter → silence"). It's deterministic, cheap, secret-free in CI, and reuses the existing STANDBY_PROPOSAL_FIXTURE replay path — minimal new machinery. The judge layer that adds noise/cost is deferred.
Proof surface
scripts/verify-proposal-eval.sh (CI anchor) + scripts/verify-live-proposal-eval.sh (gated) → JSON report in docs/evidence/ai-first-proposals/eval/.
Stop conditions
(1) First live run can't clear a sane bar → surface it as a finding (the bet is weaker than assumed); do not fudge the bar. (2) Deterministic metrics depend on a judge to be meaningful → stop, the labels are wrong. (3) Corpus needs >1 live capture pass to stabilize → stop and reconsider determinism.

Goal · Non-Goals · Constraints

Goal

Measure whether Standby's model-native proposals are useful and low-noise — and gate regressions — over a held-out labeled corpus, so the keyword-engine deletion (AGENTS.md:68) is justified by evidence rather than assertion.

Non-Goals

  • No LLM-judge faithfulness/usefulness scoring in v1. Deferred to a child (text-quality beyond span overlap). Adds nondeterminism, cost, and its own noise floor; not needed to falsify the bet.
  • Do not rebuild the model boundary. The schema, confidence floor (0.55), and evidence-citation rejection (engine.rs:641-676) are strong — measure them, don't touch them.
  • No realtime/audio provider (that's 011's downstream, tracked separately).
  • No change to the live proposal cadence/debounce (touch only if the corpus needs windowing helpers).

Constraints / invariants that must survive

  • Rust owns policy: the eval drives propose() and scores its real output — it must not bypass validation or score raw model JSON.
  • No keyword heuristics reintroduced (the verify-model-proposals.sh symbol grep stays green).
  • The existing STANDBY_PROPOSAL_FIXTURE replay path and fixture_replay.rs tests keep working unchanged.
  • No secrets or raw transcripts leak into checked-in eval reports — redact like verify-live-model-proposal.sh already does.

The request ("build an eval harness") survives challenge — but the scope is where shaping earns its keep. The failure mode to avoid is over-building a judge before the cheap deterministic signal is even in place.

Design

Corpus

≥30 labeled cases under crates/standby-core/tests/fixtures/eval/: a corpus.json manifest of cases, each:

{
  "id": "ask_paraphrased_research_01",
  "category": "ask | negation | chatter | ambiguous | operator_request",
  "transcript": [ { "id":"seg_..","speaker":"remote_1","text":"..","is_final":true,"start_ms":..,"end_ms":.. } ],
  "operator_message": null,                 // set for Ask-Standby cases
  "label": {
    "should_propose": true,
    "expected_kind": "research",            // null when should_propose=false
    "gold_evidence_ids": ["seg_.."]         // [] when should_propose=false
  },
  "recorded_response": "responses/ask_paraphrased_research_01.json"  // captured real-model output, for the CI anchor
}

Seed transcripts from real captures in docs/evidence/real-meeting/. Composition must include false-positive bait — the negation ("we should not pursue X") and chatter (small talk, status noise) categories with should_propose:false are the cases the synthetic stub fails and the ones that prove low-noise. Target a rough 60/40 positive/negative split so false-proposal-rate is well-estimated.

Grader — deterministic, decision-level (v1)

After running each case through propose(), compare the ProposalAgentDecision to the label:

MetricDefinitionCatches
False-proposal rateproposed when should_propose=false, over all negative casesnoise — the headline anti-metric; the stub fails here
RecallTP / (TP+FN) over positive casesmissed genuine asks
Precision / F1TP / (TP+FP); harmonic meanoverall decision quality
Kind accuracyon TPs, proposal.kind == expected_kindwrong action type
Evidence overlapon TPs, |cited ∩ gold| / |gold| (recall@gold)cited the wrong span — stronger than today's "id exists" check (engine.rs:641-676)

All deterministic given the model's output. No judge. The judge (faithfulness/usefulness of text) is a named child, out of v1.

What runs where

Noise floor (CLAUDE.md doctrine): the live lane runs the model K times over the corpus; each metric gets a distribution; report the bootstrap CI (resample cases) and require the lower bound to clear the bar — a point estimate inside the noise floor is not a result. The CI anchor is exact (deterministic replay), so it asserts golden metrics directly and guards the Rust pipeline (floor, evidence rejection, dedupe, pre-gates).

Build sequence — verification system first

  1. Corpus skeleton + harness before any claim. Author ~10 cases across all categories and the proposal_eval.rs runner that drives propose() and emits the metric table. This is the loop; it exists before the result.
  2. Fill the corpus to ≥30, seeded from docs/evidence/real-meeting/, weighted to include negation + chatter bait.
  3. One live capture pass (STANDBY_LIVE_MODEL=1): run the real model over the corpus, capture per-case responses → check in as recorded_response anchors, and record live metrics + bootstrap CI as the baseline + provisional bars.
  4. Wire both gates. CI anchor into verify.sh/verify-ci.sh (assert ≥ baseline); gated live script writes the redacted report. Extend the README verification table.

Pass bar / regression budget

Provisional (the first live run, step 3, calibrates these — and a failure to clear them is a finding, not a tuning exercise): false-proposal-rate ≤ 0.10 (upper CI), recall ≥ 0.80 (lower CI), kind-accuracy ≥ 0.80, evidence-overlap ≥ 0.5. CI anchor: metrics must not regress below the captured baseline.

Oracle — executable definition of done

  • ./scripts/verify-proposal-eval.sh exits 0: runs the corpus through the recorded anchor, prints precision/recall/F1/false-rate/kind/evidence-overlap, and fails if any drops below the captured baseline.
  • A deliberately corrupted case (swap a positive's transcript for chatter, or point evidence at the wrong span) makes the harness's metric move in the expected direction and, if it crosses the floor, fails the gate — proving the gate has teeth (not just "a number printed").
  • STANDBY_LIVE_MODEL=1 ./scripts/verify-live-proposal-eval.sh writes docs/evidence/ai-first-proposals/eval/report.json with per-metric point estimate + bootstrap CI over K runs, and exits non-zero if a lower CI bound is below the bar.
  • ./scripts/verify.sh stays green (the anchor is wired in); verify-model-proposals.sh heuristic-symbol grep stays green.
  • No secret or raw transcript text in the checked-in report (redacted like verify-live-model-proposal.sh).

Risks + rollout

  • Labels encode the author's bias. Mitigate: keep cases small and unambiguous; borderline cases go in an ambiguous bucket excluded from the hard bars but reported.
  • First live run undershoots the bar. That is the intended failure mode — it means the bet needs work (prompt, model, or boundary). Surface as a finding + a follow-up ticket; do not lower the bar to green. (Stop condition.)
  • Corpus drift vs model updates. The pinned model (gpt-5.5, engine.rs:16) can change behavior; the live lane re-measures, the anchor holds the line for the Rust pipeline. Re-capture anchors deliberately, with a recorded reason.
  • Rollout / undo: additive — new fixtures, one test, two scripts, README rows. Revert = drop the files + unwire from verify.sh. No product-code behavior change.

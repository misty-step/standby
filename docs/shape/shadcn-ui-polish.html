Backlog 010

Operational UI polish with shadcn-style primitives

Make Standby usable in the first working viewport: transcript on the left, current proposal/job/result on the right, honest audio status above, and meaningful side navigation.

Dependency decision

npx shadcn@latest info detects Vite/TS but no Tailwind, aliases, or components.json. Full shadcn init would add Tailwind and component plumbing to a tiny existing CSS app, so v1 uses local semantic primitives patterned after shadcn.

Outcome

Dense, calm meeting console. Running work and receipts are visible without hunting below a long proposal card.

Proof

Browser-driven desktop and mobile screenshots for idle, proposal, running/completed, failed, notes, jobs, and audio views.

Stop Rule

No text overlap, no dead nav, no console/network errors, and ./scripts/verify.sh remains green.

OptionBenefitCostVerdict
Full shadcn/Tailwind initRegistry components and future consistency.Adds Tailwind, aliases, config, and generated files before the product needs them.Defer.
Local semantic primitivesMatches current app, low churn, still uses shadcn composition ideas.No registry import path yet.Choose.
CSS-only cleanupFastest.Does not fix component hierarchy or nav/workflow issues.Insufficient.

Execution Shape

Layout. Use stable shell/view primitives and a first-viewport status rail for current job/result state.
Navigation. Keep sidenav only because Meeting, Notes, Jobs, and Audio panels now expose real state.
Audio. Make mic/system lane state and permission failures explicit; no ambiguous "live" copy.
Verification. Replace the older real-meeting screenshot set with a dedicated docs/evidence/ui-visual-qa/ packet and DOM assertions.
